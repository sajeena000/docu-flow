A Beginner's Guide to Git

Master version control and learn how to collaborate effectively on code projects.

What is Git?

Git is a Distributed Version Control System (DVCS). Unlike older centralized systems, every developer's computer has a complete copy of the project history, enabling offline work and faster operations.

It allows you to track changes, revert to previous states, and collaborate with team members without overwriting each other's work.

Why Use Git?

  • History Tracking: See who made what changes and when, with complete audit trails
  • Collaboration: Work with multiple developers simultaneously without conflicts
  • Experimentation: Try new features in isolated branches without breaking the main codebase
  • Backup: Your code is safely stored, versioned, and recoverable

Installation

Before you can use Git, you need to install it on your machine. Download the latest version from the official website for your operating system.

Download Git

  • Windows: Download installer from git-scm.com
  • macOS: Install via Homebrew: brew install git or download from git-scm.com
  • Linux: Use your package manager: sudo apt-get install git (Debian/Ubuntu) or sudo yum install git (Fedora)

Verify Installation

Terminal 
# Check if Git is installed
git --version

# Expected output:
# git version 2.x.x

First-time Setup

Set your identity so your commits are correctly attributed to you. This information will be included in every commit you make and will be visible to other collaborators.

Configure Your Identity 
git config --global user.name "Your Name"
git config --global user.email "you@example.com"

Verify Your Configuration

# View all your settings
git config --list

# View specific setting
git config user.name
git config user.email
üí° Tip: Use the same email address you use for your GitHub account to ensure your commits are properly linked to your profile and contribute to your activity graph.

Basic Commands

The core cycle of Git revolves around these essential commands. Understanding these will cover 90% of your daily Git usage and form the foundation of your version control workflow.

git init
Initialize a new local Git repository in your current directory. This creates a .git folder that tracks all changes.
git clone <url>
Download a remote repository to your local machine, including all history and branches.
git status
Check which files have been modified, added, or deleted. This is your go-to command for checking the state of your repository.
git add .
Add all changed files in the current directory to the staging area, preparing them for commit.
git add <file>
Add a specific file to the staging area. Use this for more granular control over what gets committed.
git commit -m "message"
Save a snapshot of your staged changes with a descriptive message. Write clear, meaningful commit messages!
git push
Upload your local commits to a remote server (like GitHub), making them available to other team members.
git pull
Download and merge changes from the remote repository into your current branch.

Example Workflow

# Create a new project
mkdir my-project
cd my-project
git init

# Make some changes, then:
git status                        # See what changed
git add .                         # Stage all changes
git commit -m "Initial commit"    # Save the snapshot

# Connect to GitHub and push
git remote add origin https://github.com/username/repo.git
git push -u origin main

The Staging Area

The staging area (also called the "index") is one of Git's most powerful features. It's a middle ground between your working directory and the repository, giving you precise control over what gets committed.

The Three States

  1. Working Directory: Where you edit files. Changes here are not yet tracked.
  2. Staging Area: Where you prepare commits. Files here are ready to be saved.
  3. Repository: Where commits are permanently stored. This is your project's history.

Why Stage Changes?

The staging area lets you carefully control what goes into each commit. You can:

  • Stage only related changes together, creating logical commits
  • Review changes before committing to catch mistakes
  • Create atomic commits that focus on a single task or fix
  • Exclude work-in-progress or experimental files
Working with the Staging Area 
# Add specific files
git add index.html style.css

# Add all files in a directory
git add src/

# Unstage a file (keep changes in working directory)
git reset HEAD file.txt

# View staged changes
git diff --staged

# View unstaged changes
git diff

Branching Basics

Branches are Git's killer feature. They allow you to diverge from the main line of development and work independently without affecting the stable codebase. Think of branches as parallel universes for your code!

Common Branch Commands

git branch
List all local branches. The current branch is marked with an asterisk (*).
git branch <name>
Create a new branch (but don't switch to it). The branch will be based on your current position.
git checkout <branch>
Switch to an existing branch. Your working directory will update to reflect that branch's state.
git checkout -b <name>
Create a new branch AND switch to it immediately. This is a shortcut for branch + checkout.
git merge <branch>
Merge another branch into your current branch, combining their changes.
git branch -d <name>
Delete a branch safely (only if it's been merged). Use -D to force delete.

Branch Naming Conventions

  • feature/login-page - New features or enhancements
  • bugfix/navbar-overflow - Bug fixes
  • hotfix/security-patch - Urgent fixes for production
  • docs/update-readme - Documentation updates
  • refactor/clean-api - Code refactoring
Example: Creating and Using Branches 
# Create and switch to a new branch
git checkout -b feature/dark-mode

# Make your changes and commit
git add .
git commit -m "Add dark mode toggle"

# Switch back to main
git checkout main

# Merge your feature
git merge feature/dark-mode

# Delete the feature branch (cleanup)
git branch -d feature/dark-mode

Feature-Branch Workflow

This is the standard workflow for modern development teams. It keeps the main branch clean and stable while allowing multiple developers to work on different features simultaneously without stepping on each other's toes.

The Workflow Steps

  1. Create a Branch: Start a new branch for your task or feature 
    git checkout -b feature/new-feature
  2. Develop: Make your changes and commit them locally with clear messages 
    git add .
git commit -m "Implement new feature functionality"
  3. Push: Push your branch to GitHub to share with your team 
    git push origin feature/new-feature
  4. Pull Request: Open a PR on GitHub to merge your branch into main. Add a clear description of your changes.
  5. Review & Merge: Team members review your code, suggest improvements, then approve and merge the PR when ready.
  6. Clean Up: Delete the feature branch after merging to keep your repository tidy 
    git checkout main
git pull
git branch -d feature/new-feature
üí° Best Practice: Keep your feature branches small and focused. Each branch should address one specific feature or fix. This makes code review easier and reduces merge conflicts.

Benefits of This Workflow

  • Isolation: Work on features without affecting others or breaking the main codebase
  • Code Review: Pull Requests enable peer review, catching bugs and improving code quality
  • Continuous Integration: Automated tests run before merging, ensuring stability
  • Clean History: Main branch stays stable, deployable, and easy to understand
  • Parallel Development: Multiple team members can work simultaneously without conflicts

Pull Requests

Pull Requests (PRs) are a GitHub feature that facilitates code review and collaboration. They're not part of Git itself, but they're essential to modern development workflows and team collaboration.

Creating a Pull Request

  1. Push your feature branch to GitHub
  2. Navigate to your repository on GitHub
  3. Click the "Pull requests" tab, then "New pull request"
  4. Select your feature branch to merge into main
  5. Add a descriptive title and detailed description explaining your changes
  6. Assign reviewers, add labels, and link related issues
  7. Click "Create pull request"

PR Best Practices

  • Clear Title: Use descriptive titles like "Add user authentication" or "Fix navbar overflow on mobile"
  • Detailed Description: Explain what changed, why, and any important context or decisions
  • Small Changes: Keep PRs focused and reviewable (under 400 lines when possible)
  • Tests: Include tests for new features and ensure all tests pass
  • Screenshots: Add visual proof for UI changes (before/after comparisons)
  • Link Issues: Reference related issues (e.g., "Closes #123" or "Fixes #456")
  • Self-Review: Review your own PR first to catch obvious issues

Reviewing a Pull Request

As a reviewer, you should:

  • Read the description and understand the context and purpose
  • Review the code changes carefully, looking for bugs, style issues, and improvements
  • Test the changes locally if possible to verify functionality
  • Leave constructive comments and suggestions, not just criticism
  • Approve if everything looks good, or request changes with clear explanations
Reviewing Locally 
# Fetch the PR branch (replace 123 with PR number)
git fetch origin pull/123/head:pr-123
git checkout pr-123

# Test the changes
npm test              # Run tests
npm start             # Test functionality

# Return to main when done
git checkout main
